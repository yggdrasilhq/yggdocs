# Battle Log: Service Evolution and Scope Control

Older notes show a familiar pattern: a growing wishlist of useful services.  
Mail. Sync. Media. Monitoring. Routing. Everything at once.

What improved over time was not speed of adding services.  
It was discipline in adding them.

## Pattern that scaled

- pick one service with clear user value,
- write a minimal runbook,
- add validation checks,
- define rollback,
- then move to the next service.

## Pattern that failed

- parallelizing multiple critical services,
- relying on memory instead of docs,
- shipping without failure rehearsal.

## Project rule

Scope control is not bureaucracy. It is throughput protection.

`aste cholo` (move steadily): fewer simultaneous risks, faster long-term delivery.
